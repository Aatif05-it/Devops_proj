SQL Execution Plans

Execution plans show exactly how the database engine executes a query, revealing index usage, join order, and performance bottlenecks.

Use EXPLAIN (MySQL/PostgreSQL) or SHOW PLAN (SQL Server) to analyze and optimize slow queries.

MySQL EXPLAIN basics

EXPLAIN reveals table scans, index usage, row estimates, and join strategies. 

EXPLAIN SELECT * FROM Orders o
JOIN Customers c ON o.CustomerID = c.CustomerID
WHERE o.OrderDate >= '2025-01-01';

Key columns: type (index, range, ALL), rows (estimated rows scanned), key (index used).

Reading execution plans

Look for red flags: ALL (full table scan), high rows values, Using filesort, Using temporary. 

-- BAD PLAN (needs index)
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
| id | select_type | table | type | possible_keys | key  | key_len | ref  | rows   | Extra       |
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
|  1 | SIMPLE      | orders| ALL  | NULL          | NULL | NULL    | NULL | 100000 | Using where |

-- GOOD PLAN (uses index)
|  1 | SIMPLE      | orders| ref  | idx_orderdate | idx_orderdate | 3 | const | 500 | Using index |

ref with index is much faster than ALL scanning 100K+ rows. [

SQL Server execution plans

Graphical plans show operators, costs, and estimated rows with color-coded warnings. 

-- Show actual execution plan
SET SHOWPLAN_ALL ON;
-- Or in SSMS: Ctrl+M before executing

-- Common costly operators to avoid:
-- Table Scan (100% cost)
-- Hash Match (expensive joins)
-- Sort (filesort equivalent)

Thick arrows = more rows; red text = warnings; aim to reduce total cost percentage.

Optimization using plans

Add indexes where plans show table scans; rewrite queries to leverage existing indexes. 

-- BEFORE: Slow correlated subquery
SELECT CustomerID 
FROM Orders o
WHERE NOT EXISTS (
  SELECT 1 FROM Returns r WHERE r.OrderID = o.OrderID
);

-- AFTER: Efficient LEFT JOIN (plan shows Index Seek)
SELECT DISTINCT o.CustomerID
FROM Orders o
LEFT JOIN Returns r ON o.OrderID = r.OrderID
WHERE r.OrderID IS NULL;

Execution plans turn guesswork into data-driven optimization, often yielding 10x+ speedups.
