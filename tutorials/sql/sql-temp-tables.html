SQL Temporary Tables

Temporary tables store intermediate results during complex queries or sessions, automatically cleaned up when no longer needed.

They support indexes, constraints, and transactions, making them ideal for multi-step data processing.

Session temporary tables

Tables prefixed with # (MySQL) or declared with DECLARE TABLE exist only for the current connection. 

CREATE TEMPORARY TABLE TempOrders (
  OrderID INT PRIMARY KEY,
  CustomerID INT,
  TotalAmount DECIMAL(10,2)
);

INSERT INTO TempOrders 
SELECT OrderID, CustomerID, TotalAmount 
FROM Orders 
WHERE OrderDate >= '2025-01-01';

Other connections cannot see session temp tables, and they disappear when the connection closes.

Using temp tables in complex queries

Break down multi-step calculations into readable, indexed intermediate results. 

-- Step 1: High-value customers
CREATE TEMPORARY TABLE HighValueCustomers AS
SELECT CustomerID, SUM(TotalAmount) AS LifetimeValue
FROM Orders GROUP BY CustomerID
HAVING LifetimeValue > 10000;

-- Step 2: Recent orders from high-value customers
CREATE TEMPORARY TABLE RecentHighValueOrders AS
SELECT o.*
FROM Orders o
JOIN HighValueCustomers h ON o.CustomerID = h.CustomerID
WHERE o.OrderDate >= DATE_SUB(CURDATE(), INTERVAL 30 DAY);

SELECT * FROM RecentHighValueOrders;

Temp tables with indexes perform better than subqueries or CTEs for repeated intermediate access.

Table variables (SQL Server)

DECLARE @table creates lightweight, memory-based alternatives with limited features. 

DECLARE @TopProducts TABLE (
  ProductID INT,
  ProductName VARCHAR(100),
  SalesCount INT
);

INSERT INTO @TopProducts
SELECT TOP 10 ProductID, ProductName, COUNT(*) as SalesCount
FROM OrderItems GROUP BY ProductID, ProductName
ORDER BY SalesCount DESC;

Table variables have lower overhead but lack statistics, making them suitable for small result sets.
